Fixing Malformed Card Generation in Aura

4 approaches to make Gemma's structured output reliable for lesson cards, ranked by effort and ambition.

Root Cause

Gemma 4 E2B returned a recap card with bullets: "Main idea..." (string) instead of bullets: ["Main idea..."] (array). normalizeCards() at cardGenerator.ts:42 threw immediately. The entire /generateLesson request failed — cards 1-3 (already valid) were discarded.

The JSON is syntactically valid, so jsonrepair doesn't help. The 2-retry loop in callLLMJson only retries JSON syntax errors. There are 12 hard throw sites in normalizeCards() — any single malformed field kills everything.

Gemma returns bullets: string → !Array.isArray check → throw Error → 500 response → 3 good cards lost
Approach 1 Recommended Start
Defensive Normalization + Schema Coercion
Never throw on bad shape. Coerce, then fallback.
How It Works

Replace every throw in normalizeCards() with type coercion. Add Zod schemas as the validation backbone. If coercion fails for a card, retry that single card once (not the whole lesson). Final fallback: deterministic placeholder card.

LLM output → Zod safeParse |— pass → accept |— fail → coerce (string→array, object→values, etc.) |— pass → accept (coerced) |— fail → retry 1x (temp 0.2) |— pass → accept |— fail → deterministic fallback
Coercion Examples

recap.bullets: string → split by ". " or "\n"; object → Object.values(); missing → derive from title+body

mcq.options: string[] → wrap as {id, text}; object with a/b/c → entries

story.beats: string → split by numbered patterns ("1.", "2.")

Files Changed

cardGenerator.ts (normalizeCards rewrite), new cardSchemas.ts (Zod), new fallbacks.ts

Pros
  • Zero added latency in happy path (sync coercion)
  • One bad card can't kill three good cards
  • Zod schemas = machine-readable docs for every card type
  • Foundation for all other approaches
Cons
  • Coercion is heuristic — split sentences may be awkward
  • Doesn't prevent the problem, only patches after

Approach 2
Zod Schemas + LLM Self-Repair Loop
Validate with Zod, feed errors back to Gemma for targeted self-correction.
How It Works

On Zod validation failure, format the error into a repair prompt and send it back to Gemma. The model receives its own broken output, the exact error message, and the schema — then corrects itself. Max 2 retries per card. Falls back to Approach 1 coercion.

LLM output → Zod safeParse |— pass → accept |— fail → repair prompt (broken JSON + Zod error + schema) → Gemma (temp 0.15) → Zod safeParse |— pass → accept |— fail → retry 1 more, then coerce/fallback
Repair Prompt Structure
// System: "Fix the malformed card. Change ONLY the failing fields."
// User payload:
{
  "previousOutput": { /* the broken card JSON */ },
  "validationErrors": [
    "bullets: Expected array, received string"
  ],
  "requiredSchema": { /* cardSchema() output */ }
}
Also Includes

Schema injected into original prompt system message (proactive prevention). One few-shot example per card type.

Pros
  • Repaired cards are pedagogically coherent (LLM-generated, not heuristic)
  • Zod errors give precise, targeted feedback
  • Schema in prompt prevents most failures proactively
  • Well-understood "instructor" pattern
Cons
  • 2-10 seconds extra latency per failed card (LLM round trip)
  • Small models (Gemma E2B) are weaker at self-repair than large models
  • Prompt bloat: +500-1000 tokens per card (few-shot + schema)
  • Adds queue pressure to single-job broker

Approach 3
Lightweight Agent Graph (Custom State Machine)
Per-card state machine with parallel scheduling, targeted polish, rich telemetry.
How It Works

Each card gets its own state machine: PENDING → GENERATING → VALIDATING → REPAIRING → ACCEPTED. Cards are queued in parallel via Promise.allSettled(). Failed cards loop back to repair without blocking others. Quality gate runs targeted polish on individual problematic cards instead of full-deck rewrite.

CardPipeline orchestrates per-card state machines: Card 1: PENDING → GENERATING → ACCEPTED (1.2s) Card 2: PENDING → GENERATING → ACCEPTED (1.5s) Card 3: PENDING → GENERATING → REPAIRING → ACCEPTED (3.1s) Card 4: PENDING → GENERATING → REPAIRING → COERCED (0.01s) All ACCEPTED/COERCED → quality gate → targeted polish (not full-deck)
Key Architecture

New CardPipeline class (~100 LOC state machine). generateCardsForNode() becomes a one-liner. Targeted polish replaces full-deck rewrite for most cases. Per-card telemetry exported to HUD.

Pros
  • No card blocks another — partial results survive
  • Targeted polish saves one full LLM call (~8-15s) in most lessons
  • Rich per-card telemetry for HUD
  • No external framework — extends existing broker
  • Clean extensibility for streaming, human-in-the-loop
Cons
  • More code complexity than a for-loop with try/catch
  • Broker is still single-job — no true parallelism
  • Targeted polish loses cross-card deduplication
  • Combinatorial testing surface (6 states × 12 card types)

Approach 4
Full Agentic Pipeline with LangGraph.js
Replace custom pipeline with LangGraph — graph nodes, conditional edges, state persistence, streaming.
How It Works

Entire lesson pipeline becomes a LangGraph StateGraph. Each stage is a node. Conditional edges route based on validation results. Built-in retry policies, state checkpointing (resume after crash), streaming intermediate state to frontend, and interrupt() for human-in-the-loop review.

LangGraph StateGraph: [plan] → [generate_cards] → [validate] |— all pass → [quality_gate] |— some fail → [repair] → [re_validate] |— fail → [coerce] [quality_gate] → pass → [emit] → fail → [polish] → [voice_check] → [emit] + Checkpointing, streaming, interrupt() for teacher review
Integration

Wraps existing callLLMJson() and broker in a LangChain adapter (@langchain/google-genai pointed at localhost:8080). Graph nodes call existing utility functions. May support responseSchema for constrained decoding if LiteRT-LM implements it.

Pros
  • Most extensible — adding stages is adding nodes + edges
  • State persistence — resume after crash
  • Human-in-the-loop via interrupt()
  • Streaming cards to frontend as they're validated
Cons
  • Triples dependency count (LangGraph + LangChain Core + adapter)
  • Philosophically misaligned: cloud framework for a fully-local app
  • LiteRT-LM / Gemini API compatibility gaps likely
  • 6-layer debug path vs current 3
  • Overkill: LangGraph designed for 100s of tool calls, not 6-8 cards
  • Cold start and runtime overhead

LiteRT-LM Optimizations (Apply to All Approaches)

These optimizations improve structured output regardless of which approach you choose:

  • responseSchema in generationConfig — If LiteRT-LM supports the Gemini responseSchema field, pass the JSON Schema for each card type directly. This constrains decoding at the token level — Gemma physically cannot produce bullets: string when schema says array. Test: add responseSchema to one runGeminiJob() call and verify LiteRT-LM doesn't reject it.
  • Lower temperature for structured output — Currently 0.55 for card generation. Drop to 0.35-0.4 for card types with complex schemas (dragsort, visual, flash). Keep 0.55 for creative types (story, analogy).
  • MTP (speculative decoding) with warm cache — Back-to-back card generation keeps the KV cache warm from shared system prompts. The current sequential for-loop has JS event loop overhead between iterations; queuing all cards at once eliminates gaps.
  • Prefetch resilience — Background prefetch currently fails silently when normalizeCards throws. With coercion/repair, prefetched cards succeed more often, increasing cache hit rate.

Recommended Strategy: Layer 1 Then 2

  1. Day 1 (4-6h): Implement Approach 1. Stops the crash. Adds Zod schemas as validation backbone. Introduces coercion. Every subsequent approach builds on these schemas.
  2. Day 2-3 (8-12h): Layer Approach 2 on top. Zod schemas from Approach 1 power repair prompts. Coercion becomes the fallback after repair retries fail. Defense-in-depth: schema in prompt (prevents) → LLM repair (fixes intelligently) → coercion (patches heuristically) → deterministic fallback (guarantees delivery).
  3. If product continues: Approach 3's state machine gives architectural foundation for streaming, per-card telemetry, and targeted polish. Build when sequential for-loop becomes a real bottleneck.
  4. Skip Approach 4 unless: You commit to LangGraph as an organizational choice and have confirmed full LiteRT-LM Gemini API compatibility.
